# git-repo-summarizer requirements

 Objective - Build a simple API service that takes a GitHub repository URL and returns a human-readable summary of the project

## Environment

Python 3.10+

uv environment with pyproject.toml

## POST /summarize

Accepts a GitHub repository URL, fetches the repository contents, and returns a summary generated by an LLM.

Use FastAPI.

Request body:

```
{
  "github_url": "https://github.com/psf/requests"
}
```

Field | Type | Required | Description
------ | ---- | ---- | ---- |
github_url | string | yes | URL of a public GitHub repository

Response:

```
{
  "summary": "**Requests** is a popular Python library for making HTTP requests...",
  "technologies": ["Python", "urllib3", "certifi"],
  "structure": "The project follows a standard Python package layout with the main source code in `src/requests/`, tests in `tests/`, and documentation in `docs/`."
}
```

Field | Type | Description
------ | ---- | ---- 
summary | string | A human-readable description of what the project does
technologies | string | List of main technologies, languages, and frameworks used
structure | string | Basic description of project structure

On error, return an appropriate HTTP status code and:

```
{
  "status": "error",
  "message": "Description of what went wrong"
}
```

We can test it by sending a request like:
```bash
curl -X POST http://localhost:8000/summarize \
  -H "Content-Type: application/json" \
  -d '{"github_url": "https://github.com/psf/requests"}'
```

Test:
- The server starts
- The POST /summarize endpoint exists and accepts the specified request format
- The endpoint returns a response (not an error) for a valid public GitHub repository

After starting the server, the POST /summarize endpoint should be exposed.

## LLM

LLM API key is configured via the OPENAI_API_KEY environment variable. Assume this exists in .env which is ignored.

## Handling Repo Content / Context (Core Functionality)

- Need to decide which files are important and which can be ignored (e.g., binary files, lock files, node_modules/, etc.)
- Need a strategy for fitting the most relevant information
- What gives the LLM the best understanding of a project: README? Directory tree? Key source files? Config files?

Test:
1. Functionality - The endpoint returns a meaningful, accurate summary for different repositories. Response matches the specified format (summary, technologies, structure).
2. Repo processing - Files are filtered sensibly (binary files, lock files, etc. are skipped). There is a clear strategy for selecting the most informative files.
3. Context Management - The solution handles large repositories without crashing or sending too much to the LLM. There is some strategy for staying within context limits (truncation, prioritization, summarization, etc.)
4. Prompt Engineering - The prompt(s) to the LLM are clear and produce structured, useful output.
5. Code Quality & Error Handling - Code is readable and reasonably organized. Edge cases are handled (invalid URL, private repo, empty repo, network errors). API keys are not hardcoded.
6. Documentation - README includes working setup instructions and a brief explanation of design decisions.

## README

README should contain:
- Step-by-step instructions to install dependencies and start the server (assume a clean machine with Python installed)
- Which model you chose and why (1–2 sentences is fine)
- Your approach to handling repository contents (what you include, what you skip, and why)
